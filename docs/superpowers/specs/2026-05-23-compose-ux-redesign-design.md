# HolyBean — Jetpack Compose UX 재설계 설계 문서

- 작성일: 2026-05-23
- 대상: `android/` (eloom.holybean)
- 1순위 목표: **User Flow 개선**. 모던한 Jetpack Compose 기반으로 전면 재작성.

## 1. 배경과 원칙

HolyBean은 카페 현장에서 쓰는 가로 모드 태블릿 POS 앱이다. 현재는 Fragment + XML 레이아웃 + ViewBinding/DataBinding + DrawerLayout 기반이며, MVVM(ViewModel + StateFlow), Hilt, Repository 계층은 이미 갖춰져 있다.

### 사용 빈도 (설계의 출발점)

| 화면 | 비중 | 처리 |
|------|------|------|
| 주문(Home) | **95%** | 전체화면으로 극한 최적화 — 앱의 중심 |
| 주문기록(재출력 포함) | 4.5% | 빠른 진입, 전체화면 2-pane |
| 오늘 매출 | 0.5% | 주문기록 화면 상단에 컴팩트 요약 |
| 메뉴관리 / 외상 / 기간 매출 리포트 | 거의 안 씀 | 설정(마지막 메뉴 타일) 뒤로 |

### 핵심 원칙

1. **드로어 완전 제거.** 사용자가 side drawer를 직관적으로 발견하지 못함. 자주 쓰는 기능은 모두 홈에 통합하거나 한 번의 탭으로 닿게 한다.
2. **세로 공간 절약.** 현장 태블릿은 세로가 짧다. 헤더(상단 바)를 지양하고 세로 공간을 메뉴/장바구니에 쓴다.
3. **기존 색상 기조 유지.** 오렌지 액센트(#FF7F00) / 흰 패널 / 회색 배경(#E8E9EB).
4. **기능 보존.** 주문 생성, 영수증 출력(고객용+POS용), 재출력, 삭제, 보고서 출력, 분할결제, 쿠폰 등 기존 동작을 그대로 유지하며 UX만 재설계한다.

## 2. 아키텍처

전면 재작성(빅뱅) 방식.

- **단일 Activity + Jetpack Compose Navigation.** `MainActivity` 하나에 `NavHost`. 기존 Fragment·XML 레이아웃·ViewBinding·DataBinding·DrawerLayout·`MainActivityListener`·RecyclerView 어댑터 전부 제거.
- **ViewModel/StateFlow 재사용.** 기존 `HomeViewModel`, `OrdersViewModel`, `ReportViewModel`, `MenuManagementViewModel`, `CreditsViewModel` 의 상태/이벤트 구조(`UiState` StateFlow + `UiEvent` SharedFlow)를 유지하되, Fragment 콜백 인터페이스(`OrderDialogListener` 등)에 의존하던 부분은 Compose 친화적으로 정리한다. (예: 결제 확정은 콜백 대신 ViewModel 함수 직접 호출.)
- **Hilt / Repository / Printer / Network 계층은 그대로.** `FirestoreRepository`(진행 중인 백엔드 마이그레이션 결과), `MenuRepository`, `PiPrintClient`, `HomePrinter/OrdersPrinter/ReportPrinter` 등 도메인 계층은 변경하지 않는다. 이 설계는 백엔드(AWS→Firestore) 마이그레이션과 독립적이며, 그 시점에 존재하는 Repository 인터페이스 위에 올린다.
- **의존성 정비.** Compose BOM이 매우 오래됨(`2023.03.00`) → 최신 BOM으로 상향. `org.jetbrains.kotlin.plugin.compose` 플러그인을 쓰므로 `composeOptions.kotlinCompilerExtensionVersion` 은 제거 가능. Navigation Compose, Material3 의존성 추가.

## 3. 디자인 시스템

- **컬러:** Primary 오렌지 `#FF7F00`(흰 글자), Secondary 흰 배경 + 오렌지 테두리/글자, 배경 `#E8E9EB`, 패널 흰색, 강조/합계 오렌지, 위험(삭제) 빨강 계열. 기존 `Color.kt`의 미사용 보라색 보일러플레이트는 오렌지 팔레트로 교체.
- **타이포그래피:** 기존 Pretendard 계열 폰트(extrabold/bold/semibold/medium) 유지, Compose `Typography`로 매핑.
- **공통 컴포넌트:** 메뉴 타일, 장바구니 항목, 결제수단 타일(선택형), 세그먼트 토글(컵 선택), 통계 카드(stat), 둥근 Primary/Secondary 버튼, 전체화면 오버레이 스캐폴드.

## 4. 화면 설계

### 4.1 주문 (Home) — 기본 화면

헤더 없음. 좌우 2분할.

- **좌측 메뉴 영역:** 카테고리 칩 행(전체/ICE커피/HOT커피/에이드·스무디/티·음료/베이커리) + 메뉴 그리드(4열). 각 타일은 **메뉴명 + 가격(오렌지)** 표시.
  - 그리드 **끝에 고정 타일 2개**: `쿠폰`(오렌지 점선) 과 `⚙ 설정`(회색 점선). 카테고리와 무관하게 항상 마지막에 노출.
- **우측 장바구니(폭 ~38%):**
  - 상단: `128번 주문` 라벨(좌) + `주문기록` 버튼(우, Secondary 스타일, 이모지 없음). 이 버튼이 옛 쿠폰추가 자리.
  - 항목 리스트: `메뉴명 N개` (수량은 메뉴명 옆에 자연스럽게 붙임) + 우측 정렬 금액.
  - 하단 고정: 합계(오렌지 강조) + `결제 →` Primary 버튼. 장바구니가 비면 결제 비활성화.

**상호작용**
- 메뉴 타일 탭 → 장바구니에 추가(있으면 수량 +1).
- 장바구니 항목 탭 → 수량 −1 / 0이면 제거(기존 동작 유지). (수량 조절 UX는 구현 시 +/− 명시 검토.)
- 주문번호는 진입 시 Repository에서 채번.

### 4.2 쿠폰 입력

- 메뉴 그리드의 `쿠폰` 타일 탭 → **시스템 숫자 키보드**로 금액 입력 → 장바구니에 한 줄로 추가.
- 쿠폰은 일종의 메뉴이며 금액은 **항상 합계에 더해진다(+ 부호 표기는 하지 않음)**. 장바구니에는 다른 항목과 동일하게 금액만 표시.

### 4.3 결제 — 전체화면

세로 짧은 태블릿을 위해 다이얼로그 대신 가로 전체화면. 좌우 2분할. 라벨 `128번 주문 · 결제`.

- **좌측 주문 요약:** 항목(메뉴명 N개 + 금액) 리스트 + 하단 고정 합계(오렌지). 항목이 많으면 리스트만 스크롤, 합계는 고정.
- **우측 옵션:**
  - 컵 선택: 세그먼트 토글 `일회용컵 / 머그컵`.
  - 결제 수단: 6개 타일(현금/쿠폰/계좌이체/외상/무료쿠폰/무료제공) 동등 노출, 3×2 그리드, 선택 시 오렌지 강조. (배치 순서는 중요치 않음.)
  - 주문자명: **상시 노출** 입력칸(거의 필연적으로 입력).
  - 분할결제 토글.
  - 고정 푸터: `결제 완료` Primary 버튼.

**분할결제 (가끔 사용)**
- 토글 OFF가 기본. **OFF면 우측 옵션이 한 화면에 다 들어와 스크롤이 없어야 한다.**
- 토글 ON → 2번째 결제 수단이 나타남. 2번째 수단 후보는 **{계좌이체, 현금, 쿠폰, 무료쿠폰}** 이며, 1번째로 고른 수단을 제외한 **나머지 3개를 1줄로** 렌더. 금액 입력 시 1번째 수단이 자동으로 잔액을 가짐.
- 늘어난 부분(분할/주문자명)으로 화면을 넘길 때만 옵션 영역이 내부 스크롤되고, `결제 완료` 버튼과 좌측 요약은 고정.

**규칙(기존 로직 유지)**
- `계좌이체`·`외상` 포함 시 주문자명 필수.
- `무료제공` 선택 시 분할결제 비활성화.
- 분할 금액은 잔액 > 0 이어야 함.
- `외상` 포함 시 creditStatus 기록.
- 확정 시: 주문을 Repository에 저장 → 홈으로 복귀, 영수증(고객용+POS용, 컵 옵션 포함)을 ApplicationScope에서 출력.

### 4.4 주문기록 + 오늘 매출 — 전체화면

장바구니의 `주문기록` 버튼으로 진입. 상단 컴팩트 매출 스트립 + 2-pane.

- **상단 매출 스트립(컴팩트, 상시):** `오늘 총 판매` · `총 건수` · `총 잔수` + `보고서 출력` 버튼.
  - 총 잔수 = 당일 음료 항목 수량 합계(쿠폰 등 비음료 제외) — 기본 정의, 리뷰 시 확정.
  - 결제수단별·메뉴별 세부는 화면에 두지 않고 **출력물에서 확인**. `보고서 출력`은 현재 리포트 출력과 동일 동작(오늘 기준). 날짜 범위 선택은 설정의 기간 매출 리포트로 분리.
- **좌측 목록:** 오늘 주문(최신 상단). 각 항목 `N번 / 금액 / 주문자 / 결제수단`. **검색 없음**, 스크롤 탐색. 첫 항목 자동 선택(기존 동작).
- **우측 상세:** `N번 주문` + 항목(메뉴명 N개 + 금액) + 합계 + `재출력`(Primary) / `삭제`(빨강 Secondary). 주문자·결제수단은 우측에 표시하지 않음(좌측 목록에 이미 있음).
- 삭제는 확인 다이얼로그 후 수행, 성공 시 목록 갱신.
- 닫기 → 주문(홈) 복귀.

### 4.5 설정 — 마지막 메뉴 타일로 진입

- 메뉴 그리드 끝의 `⚙ 설정`(회색) 타일 → 설정 시트.
- 시트 항목:
  - **메뉴 관리** (비밀번호 보호) — 기능 그대로 Compose 포팅 + 새 테마.
  - **외상 관리** — 기능 그대로 포팅 + 새 테마.
  - **기간 매출 리포트** (날짜 범위 선택 + 출력) — 기능 그대로 포팅 + 새 테마.
  - **개발자 도구** (아래).

### 4.6 개발자 도구 — 현장 즉시 진단

빨간불이면 현장에서 바로 조치할 수 있게.

- Pi 프린터 `/health` 핑 — 정상 여부 + 응답 시간.
- 네트워크 연결 상태 — Wi-Fi / 프린터 서버 IP.
- Firestore 연결 상태.
- 프린터 서버 URL 표시.
- **테스트 영수증 출력** 버튼.

### 4.7 희귀 화면 (메뉴관리·외상·기간리포트)

기능은 현재와 동일. XML→Compose로 옮기고 오렌지 기조만 입힌다. 재설계·기능 추가 없음. 추후 '필요시' 웹 백오피스로 이전 가능(이번 범위 밖).

## 5. 내비게이션 맵

```
주문(홈, 기본)
 ├─ [결제 →]      → 결제 (전체화면)         → 확정 시 홈 복귀
 ├─ [주문기록]    → 주문기록+오늘매출 (전체화면) → [보고서 출력]
 └─ [⚙ 설정 타일] → 설정 시트
                     ├─ 메뉴 관리 (비밀번호)
                     ├─ 외상 관리
                     ├─ 기간 매출 리포트
                     └─ 개발자 도구
```

드로어 없음. 모든 진입은 홈에서 한 번의 탭.

## 6. 범위 밖 (Out of Scope)

- 웹 백오피스(미래 '필요시').
- 백엔드 변경(AWS→Firestore 마이그레이션은 별도 진행 중 작업).
- 메뉴관리/외상/기간리포트의 기능 재설계(이번엔 테마 포팅만).

## 7. 테스트 / 검증

- 도메인·ViewModel 재사용 → 기존 단위 테스트(Home/Orders/Report/MenuManagement/Credits ViewModel, Printer, Repository, Aggregation) 가능한 한 유지. 콜백→직접호출 리팩터링에 맞춰 수정.
- 핵심 플로우(주문 담기→결제→확정, 쿠폰 입력, 분할결제, 재출력, 삭제)에 대한 Compose UI 테스트 추가.
- 현장 회귀: 영수증 2종 출력, 보고서 출력이 기존과 동일 산출물인지 확인.

## 8. 리스크 / 주의

- **빅뱅 재작성** — 한 번에 큰 변경. 구현 계획에서 화면 단위로 쪼개 검증 체크포인트를 둔다(주문→결제→주문기록→설정/희귀화면 순 권장).
- Compose BOM 상향에 따른 Material3 API 변동.
- 진행 중인 Firestore 마이그레이션과의 동시 작업 — Repository 인터페이스 기준으로 결합도 최소화.
- 가로 고정 + `windowFullscreen`(시스템 바 숨김) 등 기존 설정 유지.
