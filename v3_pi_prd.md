# PRD - 영수증 프린트 서버

## 1. 배경 및 문제 정의

### 현재 상황
Android 앱에서 영수증 프린터로 출력 시 **Bluetooth 연결을 사용**하고 있다.

### 문제점
- Bluetooth 페어링이 자주 끊김
- 재연결 처리가 복잡하고 실패 원인 파악이 어려움
- 거리/간섭에 민감해 출력 안정성이 낮음
- 출력 속도가 느림

### 영향
영수증 출력 실패는 매장 운영에 직접적인 차질을 유발한다.
주문 처리가 지연되고, 직원이 수동으로 재시도해야 한다.

---

## 2. 목표

### 핵심 목표
Bluetooth를 **유선 기반 네트워크 출력 구조로 대체**하여 출력 안정성을 확보한다.

### 성공 기준
| 지표 | 현재 (Bluetooth) | 목표 |
|------|-----------------|------|
| 출력 성공률 | 불안정 | 99% 이상 |
| 출력 실패 시 자동 복구 | 수동 재시도 | 자동 재시도 3회 |
| 연결 끊김 빈도 | 잦음 | 거의 없음 |
| 출력 지연 | 느림 | 1초 이내 |

### 비목표 (Out of Scope)
- **AWS 비즈니스 로직의 Pi 이전** (주문/외상/매출/메뉴 처리는 기존대로 Android↔AWS 직통 유지)
- Pi의 오프라인 주문 접수 / 로컬 채번 / source-of-truth 이전
- 여러 매장 통합 관리
- 출력 이력 장기 보관 및 분석
- 프린터 외 다른 주변기기 제어

> **비목표 보충 — Pi는 AWS 게이트웨이가 아니다.**
> Pi는 인쇄만 책임지는 stateless 서버이며, AWS로의 통신은 NAT 라우팅으로 *경유*만 시킨다.
> AWS 로직을 Pi로 옮기는 안(프록시 포함)은 인쇄 경로와 AWS 경로를 한 프로세스에 결합시켜
> "영수증 출력"이라는 최우선 가치를 오히려 위협하므로 채택하지 않는다. (자세한 근거는 §4.1)

## 3. 사용자 및 유즈케이스

### 사용자
- 매장 직원 (Android 앱 사용자)
- 시스템 관리자 (초기 설치 및 유지보수)

### 핵심 유즈케이스
1. **주문 영수증 출력** - 직원이 주문 완료 시 영수증을 출력한다
2. **출력 실패 인지** - 출력이 실패하면 직원이 즉시 알 수 있어야 한다
3. **자동 복구** - 일시적 오류는 시스템이 스스로 재시도한다
4. **무중단 운영** - 장비 재부팅 후 별도 조작 없이 출력 가능 상태로 복귀한다

## 4. 솔루션 개요

### 구조
```
                          ┌─ Wi-Fi(핫스팟) ─┐
Android App ──────────────┤                 │
  │                       │   Raspberry Pi   │──USB──▶ 영수증 프린터
  │  ① 인쇄: POST /print   │ (핫스팟+NAT+      │
  │     (Pi 프린트 서버)    │  프린트 서버)     │
  │                       │                 │
  └─ ② 주문/데이터: AWS ────┴── NAT 라우팅 ────┴──이더넷──▶ 인터넷 ──▶ AWS API GW/Lambda
       (Android→AWS 직통, Pi는 경유만)
```

- **① 인쇄 경로** — Android → Pi 프린트 서버. AWS를 전혀 모르는 격리된 경로.
- **② 데이터 경로** — Android → (Pi NAT 경유) → AWS. 기존 Android 로직 그대로.
- 두 경로는 **독립적**이다. AWS가 느리거나 끊겨도 인쇄는 영향받지 않는다.

### 핵심 결정 사항
| 항목 | 결정 | 근거 |
|------|------|------|
| 출력 방식 | Wi-Fi HTTP (Bluetooth 대체) | 안정성, 재시도 용이 |
| 네트워크 | Pi 자체 Wi-Fi 핫스팟 | 공유기 환경에 무관하게 동작 |
| **인터넷 경로** | **Pi가 이더넷 업링크 + NAT 라우팅** | **핫스팟에 붙은 채로 Android가 AWS 직통** |
| **책임 분리** | **Pi=인쇄 전용(stateless), Android=AWS 직통** | **인쇄 경로 격리 → 최우선 가치(영수증) 보호** |
| 중계 장비 | Raspberry Pi 3B | 이더넷/USB 다수, 안정적 |
| 서버 언어 | **Rust (clippy 엄격 적용)** | **ESC/POS 명령을 합타입으로 모델링 → 컴파일 타임 정확성 보장. "한 번의 견고한 구현" 부합** |
| **ESC/POS 생성 위치** | **Pi (Android는 구조화 JSON 전송)** | **포맷을 앱 업데이트 없이 Pi 배포만으로 변경 가능. 명령 집합 모델링은 Rust enum의 강점** |
| 프린터 | 세우 SLK-TS400B | ESC/POS 호환, Linux 지원 |
| 프로세스 관리 | systemd | 부팅 자동 시작 + 크래시 자동 복구 |

### 동작 흐름
1. Android 앱이 Pi 핫스팟에 Wi-Fi로 연결 (이 시점부터 인터넷은 Pi NAT 경유)
2. 주문 생성 시 AWS와 통신 (`/ordernum`, `POST /order` — 기존 로직, Pi NAT 경유)
3. 인쇄는 **AWS 통신과 독립적으로** Pi 프린트 서버에 **구조화 JSON**으로 요청 (`POST /print`)
4. 출력 전 서버 상태 확인 (`/health`)
5. 인쇄 실패 시 최대 3회 자동 재시도
6. Pi가 JSON을 ESC/POS 명령으로 생성하고, 잡 큐에 직렬화해 USB로 프린터에 전달

> **인쇄 요청 형식** — Android는 ESC/POS 바이트가 아니라 구조화된 명령 배열을 보낸다.
> Pi가 이를 `PrintCommand` 합타입으로 역직렬화해 ESC/POS로 변환한다. 예:
> ```json
> { "commands": [
>     { "type": "text", "content": "홀리빈", "bold": true, "align": "center" },
>     { "type": "divider" },
>     { "type": "text", "content": "아메리카노  4,000" },
>     { "type": "cut" }
> ] }
> ```

### 4.1 책임 분리 근거 (왜 AWS 로직을 Pi로 옮기지 않는가)

현재 Android는 주문 채번·생성·조회, 외상, 매출, 메뉴를 **AWS(API Gateway+Lambda)** 에 의존하며,
주문번호는 **AWS가 단일 발급**한다. 로컬 DB(Room)는 메뉴 캐시뿐이다.

AWS 로직을 Pi로 옮기는 안의 트레이드오프:

| 기준 | A. AWS 로직 Pi 이전(프록시) | **B. 책임 분리 (채택)** |
|------|------|------|
| 구현 비용 | 10개 엔드포인트 Go 재구현 | 기존 Android 로직 유지, 인쇄 호출만 교체 |
| 인쇄 경로 격리 | **AWS 코드와 한 프로세스에 결합** | **완전 격리 (AWS 무관)** |
| 장애 범위 | Pi 죽으면 인쇄+데이터 전부 다운 | Pi 죽어도 데이터는 안전, 인쇄만 영향 |
| 주문번호 정합성 | 분산 채번 문제 발생 | AWS 단일 발급 유지 |
| 오프라인 이점 | 프록시 수준이면 **이점 없음** | 동일 (해당 없음) |

→ 프록시 수준 이전은 **비용만 있고 이점이 없으며, 최우선 가치인 인쇄 경로를 AWS와 결합**시킨다.
오프라인 접수가 실제로 필요해질 경우(인터넷 잦은 단절 관측 시) 별도 안건으로 재검토한다.

## 6. 제약 및 가정

### 제약
- Android 기기는 Pi 핫스팟에 연결되어 있어야 함
- Pi는 이더넷 업링크가 연결되어 있어야 하며, NAT 라우팅이 설정되어야 함
- 인터넷(AWS) 가용성은 Pi 업링크 상태에 종속됨 (단절 시 인쇄는 가능, 주문 생성은 불가)
- 배포 시 개발 PC도 Pi 핫스팟에 연결 필요

### 가정
- 4년간 운영으로 유즈케이스와 엣지 케이스가 명확히 파악됨
- 빠른 반복 배포보다 한 번의 견고한 구현이 더 중요함
- Pi와 프린터는 물리적으로 같은 장소에 고정 설치됨

## 7. 마일스톤

| 단계 | 내용 |
|------|------|
| 1 | Pi 핫스팟 + NAT 라우팅 구성 (Android가 핫스팟 경유로 AWS 도달 확인) |
| 2 | Rust 프린트 서버 구현 (`/print`, `/health`, `PrintCommand` 모델 + ESC/POS 변환, 잡 큐 직렬화) |
| 3 | 프린터 USB 연동 및 ESC/POS 출력 검증 |
| 4 | systemd 등록 + 자동 복구 검증 + ARM 크로스 컴파일(`cross`) 배포 파이프라인 |
| 5 | Android 인쇄 경로 교체 (Bluetooth → Pi HTTP, 구조화 JSON 전송 + 재시도). AWS 로직은 무변경 |
| 6 | 통합 테스트 (인쇄 경로와 AWS 경로 독립성 검증 포함) 및 운영 배포 |
